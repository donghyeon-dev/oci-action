Section 01왜 GitHub Actions로 옮기는가

Cloud Shell 자동화는 단기엔 빠르게 시작할 수 있지만, 길어질수록 다음 한계에 부딪힌다.

항목Cloud ShellGitHub Actions
세션 유지 20분 비활성 → 종료, 누적 24시간 → 강제 종료 ✅ 영구. 매 cron마다 새 컨테이너
IP 차단 위험 동일 IP에서 반복 시도 → silent block ✅ 매번 다른 GitHub Runner IP
실행 빈도 수동 조정 필요 cron 한 줄로 영구 스케줄
실패 추적 로그 파일 직접 봐야 함 Actions 탭에 모든 실행 기록
비용 무료 Public repo 무료, Private 월 2000분

Cloud Shell은 "내가 켜둔 만큼 돈다." GitHub Actions는 "내가 잠들어도 돈다."

현재 상황의 핵심 문제

Cloud Shell에서 30초 간격으로 누적 시도한 결과, OCI가 launch 요청을 silent block 중이다 (응답 무반응). 이 상태에서 동일 환경으로 계속 시도하면 차단이 더 강화될 뿐. 환경 자체를 바꾸는 것이 가장 효과적인 해결.

핵심 설계 원칙

  • 매 실행은 독립적: 컨테이너가 매번 새로 생성, 상태는 OCI 자체에서만 조회
  • 중복 방지: 매번 먼저 인스턴스 존재 여부 확인. 있으면 즉시 종료

약 15분

PHASE 03

핵심 워크플로우 작성

약 20분

.github/workflows/launch.yml 작성. cron 15분 스케줄, OCI 공식 Action 사용, 인스턴스 존재 확인 → launch 시도 → 결과 알림.

PHASE 04

안전장치 추가

가장 중요한 단계. 한도 초과 시 자동 정지, 성공 시 cron 비활성화, 일일 시도 제한, 비용 모니터링.

Phase 01OCI API 키 발급

1-1. API Key 페어 생성

  1. OCI Console 우측 상단 프로필 아이콘 → My profile
  2. 좌측 Resources → API keys → Add API key
  3. "Generate API key pair" 선택
  4. Download private key 클릭 → .pem 파일 다운로드 (이름 예: oracleidentitycloud_donghyeondev-04-29-XX.pem)
  5. Add 클릭

1-2. Configuration File Preview에서 4가지 정보 복사

Add 직후 화면에 표시되는 미리보기:

[DEFAULT]
user=ocid1.user.oc1..xxxxxxxxxxxxxx
fingerprint=12:34:56:78:90:ab:cd:ef:...
tenancy=ocid1.tenancy.oc1..xxxxxxxxxxxxxx
region=ap-chuncheon-1
key_file=<path to your private keyfile>

이 4가지 값을 메모장에 복사해두세요. 이 페이지를 닫으면 fingerprint는 다시 볼 수 없으나, API keys 목록 페이지에서 다시 확인 가능합니다.

1-3. Private Key 파일 내용 확보

다운받은 .pem 파일을 메모장(또는 VS Code)으로 열기. 내용은 다음과 같이 보입니다:

-----BEGIN PRIVATE KEY-----
MIIEvQIBADANBgkqhkiG9w0BAQEFAASCBKcwggSjAgEAAoIBAQC3...
...여러 줄...
...AVCx2hYYqq3X2HNBQ==
-----END PRIVATE KEY-----

전체 내용을 통째로 복사해서 별도 메모장에 보관. -----BEGIN부터 -----END PRIVATE KEY-----까지 한 글자도 빠짐 없이.

보안 경고

이 .pem 파일은 OCI 계정의 인증 자격증명입니다. 절대로 GitHub에 직접 커밋하지 마세요. 반드시 GitHub Secrets에만 저장. 만약 실수로 노출되면 즉시 OCI Console에서 해당 API key를 삭제하고 새로 발급.

Phase 03핵심 워크플로우 작성

3-1. 워크플로우 파일 생성

GitHub 레포 페이지에서:

  1. Add file → Create new file
  2. 파일명: .github/workflows/launch.yml (슬래시 그대로 입력하면 폴더 자동 생성)

3-2. 워크플로우 내용

name: Launch OCI A1 Instance

on:
  schedule:
    # 매 15분마다 (UTC 기준). 너무 자주는 차단 위험.
    - cron: '*/15 * * * *'
  workflow_dispatch:  # 수동 트리거 가능

jobs:
  launch:
    runs-on: ubuntu-22.04
    timeout-minutes: 5
    env:
      OCI_CLI_USER: ${{ secrets.OCI_CLI_USER }}
      OCI_CLI_TENANCY: ${{ secrets.OCI_CLI_TENANCY }}
      OCI_CLI_FINGERPRINT: ${{ secrets.OCI_CLI_FINGERPRINT }}
      OCI_CLI_KEY_CONTENT: ${{ secrets.OCI_CLI_KEY_CONTENT }}
      OCI_CLI_REGION: ${{ secrets.OCI_CLI_REGION }}

    steps:
      - name: Check existing instance
        id: check
        uses: oracle-actions/run-oci-cli-command@v1.3.0
        with:
          command: |
            compute instance list \
              --compartment-id ${{ secrets.OCI_COMPARTMENT_ID }} \
              --lifecycle-state RUNNING \
              --query "data[?shape=='VM.Standard.A1.Flex'] | length(@)" \
              --raw-output

      - name: Skip if instance exists
        if: steps.check.outputs.output != '0' && steps.check.outputs.output != ''
        run: |
          echo "Instance already exists. Skipping launch."
          curl -s -X POST "https://api.telegram.org/bot${{ secrets.TELEGRAM_BOT_TOKEN }}/sendMessage" \
            -d chat_id="${{ secrets.TELEGRAM_CHAT_ID }}" \
            -d text="Instance already running. Disable workflow."
          exit 0

      - name: Try launch instance
        id: launch
        if: steps.check.outputs.output == '0' || steps.check.outputs.output == ''
        continue-on-error: true
        uses: oracle-actions/run-oci-cli-command@v1.3.0
        with:
          command: |
            compute instance launch \
              --availability-domain "${{ secrets.OCI_AVAILABILITY_DOMAIN }}" \
              --compartment-id ${{ secrets.OCI_COMPARTMENT_ID }} \
              --shape VM.Standard.A1.Flex \
              --shape-config '{"ocpus":1,"memoryInGBs":6}' \
              --image-id ${{ secrets.OCI_IMAGE_ID }} \
              --subnet-id ${{ secrets.OCI_SUBNET_ID }} \
              --display-name dh-server-a1 \
              --assign-public-ip true \
              --metadata '{"ssh_authorized_keys":"${{ secrets.OCI_SSH_PUBLIC_KEY }}"}'
          silent: false

      - name: Notify on success
        if: steps.launch.outcome == 'success'
        run: |
          curl -s -X POST "https://api.telegram.org/bot${{ secrets.TELEGRAM_BOT_TOKEN }}/sendMessage" \
            -d chat_id="${{ secrets.TELEGRAM_CHAT_ID }}" \
            -d text="🎉 A1 instance launched! Check OCI Console."

      - name: Log on failure (no notify)
        if: steps.launch.outcome == 'failure'
        run: |
          echo "Launch attempt failed (likely 'Out of capacity'). Will retry in 15 min."
설계 포인트
  • 실패 시 알림 안 보냄: 매번 "자원 부족" 알림 오면 폭격 됨. 성공 시에만 알림.
  • 인스턴스 이미 있으면 즉시 종료: 한도 초과 방지의 첫 번째 안전장치.
  • continue-on-error: launch 실패해도 워크플로우 자체는 성공 처리. cron 계속 돔.
  • timeout-minutes: 5: 무한 대기 방지. silent block 걸려도 5분 후 자동 종료.

3-3. 커밋

페이지 하단 commit message: "Add launch workflow" → Commit changes

Phase 04안전장치 추가

이게 정말 중요합니다. 안전장치 없이 cron 돌리면 의도치 않은 결과가 누적될 수 있어요.

4-1. 일일 한도 — 워크플로우 자동 비활성화

실수로 한도 초과 같은 명백한 에러가 반복되면 워크플로우를 자동 비활성화. 위 워크플로우의 마지막 step 추가:

      - name: Disable on limit exceeded
        if: contains(steps.launch.outputs.output, 'LimitExceeded')
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          gh workflow disable launch.yml -R ${{ github.repository }}
          curl -s -X POST "https://api.telegram.org/bot${{ secrets.TELEGRAM_BOT_TOKEN }}/sendMessage" \
            -d chat_id="${{ secrets.TELEGRAM_CHAT_ID }}" \
            -d text="⚠️ Workflow disabled: LimitExceeded. Check existing instances in OCI."

4-2. 워크플로우 권한 추가

위 step이 자기 자신을 비활성화하려면 권한이 필요합니다. jobs.launch 아래에:

    permissions:
      actions: write
      contents: read

4-3. 성공 시 자동 비활성화

인스턴스가 잡히면 더 이상 cron 돌릴 필요가 없습니다. 4-1과 같은 패턴으로 성공 시에도 비활성화:

      - name: Disable workflow on success
        if: steps.launch.outcome == 'success'
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: gh workflow disable launch.yml -R ${{ github.repository }}

4-4. 통계 알림 (옵션)

매일 오전 9시에 어제 시도 횟수 요약 알림:

# 별도 워크플로우 .github/workflows/daily-summary.yml
on:
  schedule:
    - cron: '0 0 * * *'  # UTC 0시 = KST 9시
⚠ 가장 흔한 실수

안전장치 없이 cron만 걸어두고 잊어버리면, 인스턴스가 잡힌 후에도 매 15분마다 추가 시도가 계속됩니다. 결국 한도 초과 → 봇이 망가짐. "성공 시 자동 비활성화"는 필수입니다.

Phase 05테스트 & 가동

5-1. 수동 트리거로 첫 실행

  1. 레포 → Actions 탭
  2. 좌측 "Launch OCI A1 Instance" 워크플로우 선택
  3. 우측 Run workflow 버튼 → Run workflow
  4. 30초 정도 후 새 실행이 목록에 나타남
  5. 클릭해서 로그 확인

5-2. 검증 체크리스트

  • 인증 step 통과 (오타나 권한 문제 없음)
  • "Check existing instance" step에서 숫자 (보통 0) 출력
  • "Try launch instance" step 실행됨
  • 결과: Out of host capacity 또는 성공 (둘 다 정상)
  • 워크플로우 전체는 ✅ 녹색으로 끝남 (continue-on-error 덕분)

5-3. cron 자동 실행 확인

15분 후 Actions 탭에서 자동 실행이 또 한 번 보이면 정상. 그대로 두고 텔레그램 기다리면 됨.

5-4. 흔한 첫 실행 에러

에러원인해결
NotAuthenticated Secret 중 하나 잘못됨 OCI_CLI_FINGERPRINT, OCI_CLI_KEY_CONTENT 재확인
InvalidPrivateKey .pem 내용이 잘림 BEGIN~END 한 글자도 빠짐없이 다시 붙여넣기
NotFound: subnet SUBNET_ID 잘못 OCI Console에서 다시 복사
silent block (timeout) OCI 측 차단 아직 안 풀림 몇 시간 더 기다리고 다시 트리거

Phase 06성공 후 마무리

6-1. 잡혔다는 알림이 오면

  1. OCI Console → Compute → Instances → dh-server-a1 확인
  2. 상태가 RUNNING이면 성공
  3. Public IP 메모
  4. 워크플로우는 자동 비활성화되어 있을 것 (Phase 4-3 덕분). 안 됐으면 수동 비활성화: Actions 탭 → 워크플로우 → ⋯ 메뉴 → Disable workflow

6-2. SSH 접속 테스트

# Windows PowerShell
ssh -i $HOME\.ssh\oci_a1 ubuntu@<public-ip>

6-3. OCPU/메모리 확장 (선택)

1 OCPU/6GB로 잡았다면 콘솔에서 4 OCPU/24GB로 확장 가능:

  1. 인스턴스 페이지 → Edit
  2. OCPU: 4, Memory: 24 입력
  3. 저장 → 인스턴스 자동 재부팅

6-4. 기존 OpenClaw 가이드로 복귀

이전 가이드의 Section 04: 서버 초기 설정부터 이어서 진행. Tailscale, Docker, OpenClaw 순으로.

Section 04리스크 & 대처

리스크 1: GitHub 계정 차단

Cron 빈도가 너무 높으면 GitHub 측에서 abuse로 판단할 수 있음. 15분 간격은 안전 범위. 5분 이하는 위험.

리스크 2: OCI 측 차단 누적

Cloud Shell에서 이미 누적된 차단이 GitHub IP에도 영향을 줄 수 있음. 차단이 풀린 후 (최소 12시간 이후) 첫 실행 권장.

리스크 3: API 키 노출

실수로 .pem 내용이나 OCID를 코드에 직접 박는 일이 있음. 모든 민감 정보는 Secret에만. .gitignore에 *.pem, *.env 추가.

리스크 4: 한도 초과로 비용 발생

Always Free 한도 초과한 인스턴스가 만들어지면 시간당 과금. OCPU=1, MEMORY=6 고정으로 한도 절대 초과 안 함. 추가 안전 장치로 OCI Budget 알림 설정도 권장.

리스크 5: 텔레그램 토큰 노출

이번 작업 중 토큰이 두 번 노출됐으니 새 토큰 발급한 그대로 Secret에. 노출되면 누군가 봇으로 메시지 보낼 수 있음 (계정 탈취는 아님).
